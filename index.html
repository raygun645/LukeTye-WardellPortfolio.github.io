Procedural Level Generation

Masters Project 1

Role Solo Developer
Engine Unreal Engine 5
Focus PCG Algorithms
Duration 12 Weeks

The Challenge

The core objective was to expand upon my undergraduate research into Procedural Content Generation (PCG). I needed to determine which algorithms were most viable for creating valid, playable layouts for a dungeon crawler, ensuring that levels were not just random, but structured and navigable.

The Solution

I researched and implemented multiple generation techniques and conducted quantative testing with volunteers to see which was most viable amongst testers. This culminated in a single modular system being selected, capable of generating a conected dungeon level. The final systems successfully developed were:

  • Cellular Automata: Used to create organic, cave-like structures.
  • Binary Space Partitioning (BSP): Used to create structured rooms within a level.
  • Depth-First search: Used to create small interconected levels with rooms.

Technical Outcomes

3 Distinct Algorithms Developed
<1s Generation Time per Level

Key Learning: This project formed the backbone for my final Masters project. It taught me the importance of seeding in RNG to ensure bugs could be reproduced and fixed reliably.

Procedural Asset & Placement Systems

Masters Project 2

Role Solo Developer
Engine Unreal Engine 5
Focus Asset Pipeline Tooling
Duration 12 Weeks

The Challenge

Following my previous project on level generation, I faced a new problem: Empty Rooms. The challenge was twofold: I needed systems to procedurally generate the content found within the levels (weapons, items, enemies) automatically, and I needed an intelligent placement algorithm to ensure these objects didn't spawn inside walls or inaccessible areas.

The Solution

I developed two distinct subsystems to handle Content and Context:

  • Modular Asset Generation: A "Constructor" script that assembles unique weapons/enemies/loot by randomly combining compatible meshes (e.g., Stock + Barrel + Grip) and assigning stats based on the resulting mesh map.
  • Context-Aware Placement: A validation pass that checks the generated level. It uses grid-checks to identify valid "Floor" surfaces before spawning enemies or loot

Technical Outcomes

3 Asset Types (Weapons, Items, Enemies)
100% Valid Spawn Placement

Key Learning: This project highlighted the difference between "Random" and "Procedural." I learned that purely random values create noise; defining Constraints (rules for what parts fit together) is what creates usable game content.

AI Decision Making Architectures

Masters Project 3

Role Solo Developer
Engine Unreal Engine 5
Focus Behavior Systems
Duration 12 Weeks

The Challenge

With a procedural level and asset system in place, the final piece was intelligent agents that could navigate it. The core challenge was determining which AI architecture offered the best balance of performance vs. complexity. I needed an agent that could handle the unpredictability of a procedural map without breaking its behavior loop.

The Solution

I conducted a comparative implementation study, building three distinct AI controllers to test their viability in the dungeon crawler environment:

  • Finite State Machines (FSM): Implemented for simple enemies (e.g., Patrol → Chase → Attack). Efficient, but lacked nuance.
  • Fuzzy Logic (FuSM): Added to introduce "uncertainty," allowing enemies to be in partial states (e.g., "Cautiously Aggressive") rather than binary on/off states.
  • Behavior Trees (BTs): The winning solution. This hierarchical approach allowed for modular decision-making (Decorators, Selectors, Sequences) that scaled best with the complex procedural environment.

Technical Outcomes

3 Architectures Built from Scratch
BTs Selected as Optimal Solution

Key Learning: While Finite State Machines are easier to set up, they become "spaghetti code" as complexity grows. Behavior Trees, while harder to implement initially, provided the modularity needed to separate the Data (what the AI knows) from the Logic (what the AI does).

Integrated Dungeon Crawler Prototype

Masters Final Project - Encapsulation

Role Solo Developer
Engine Unreal Engine 5
Focus System Architecture
Status Playable Build (Development Ongoing)

The Challenge

The final hurdle was Integration. I had three different systems, Level Generation, Asset Placement, and AI that worked in isolation but needed to function cohesively in real-time. The challenge was to orchestrate the "Initialization Order" so the AI wouldn't spawn before the NavMesh was built, and loot wouldn't spawn before the level existed.

The Solution

I developed a Level Manager Architecture to handle the generation lifecycle and implemented core gameplay mechanics to turn the tech demo into a game:

  • Lifecycle Management: Created an event-driven initialization pipeline (Generate valid Level -> Spawn valid Assets -> Activate AI).
  • Gameplay Loops: Implemented a combat system, inventory management, and a UI layer to provide context for the procedural elements.
  • User Validation: Conducted playtesting sessions with volunteers to gather qualitative data on "Game Feel" and system stability.

Technical Outcomes

4 Systems Integrated Successfully
Stable Runtime Performance

Key Learning: This project taught me that "Technical Complexity" does not always equal "Fun." The playtests revealed that while the underlying generation was impressive, the pacing of the gameplay required manual tuning variables to ensure the procedural difficulty curve felt fair.

Farmer: Economy & Grid Systems

Personal Project - In Development

Role Solo Developer
Engine Unreal Engine 5
Focus BP to C++ Workflow
Status Ongoing

The Challenge

The goal of "Farmer" is to create a farming game like others present in the current market. The technical challenge was to create a scalable interaction system where the player can manipulate the world (plowing, planting, harvesting) on a grid-based layout. I also wanted to practice a professional Rapid Prototyping Workflow: validating mechanics quickly without getting bogged down in syntax initially.

The Solution

I implemented a strict "Prototype-First" methodology:

  • Phase 1 (Visual Scripting): All mechanics (Crop Growth, Inventory, Shop Interface) are first built in Blueprints. This allows for instant iteration and debugging of the "Game Feel."
  • Phase 2 (C++ Refactoring): Once a mechanic is fun and stable, I rewrite the core logic in C++. This moves heavy computation off the game thread and exposes clean, optimized functions back to the Editor.

Technical Outcomes

Hybrid C++ Logic / BP Visuals
Grid State-Based Plot System

Key Learning: This project improved my understanding of UInterfaces. This allows the Player to execute generic commands (e.g., "Use Tool") that result in unique behaviors depending on the target (Soil, Crop), removing the need for complex casting or branching logic.

Machine Learning: Autonomous Agent

Personal Research Project

Role Solo Developer
Engine Unreal Engine 5
Tech Reinforcement Learning
Status Prototype

The Challenge

I wanted to explore how Machine Learning (ML) could be integrated into a game engine environment. The goal was to train an agent to navigate a track autonomously. The core challenge was defining the inputs (vision) and outputs (steering) so the agent could "perceive" the track boundaries in real-time.

The Solution

I implemented a Neural Network based on Reinforcement Learning principles:

  • Sensory Input (Track Spline): The car attempts to follow a track spline. These values are normalized and fed into the network.
  • Action Output: The network outputs throttle and steering values.
  • The Reward Function: I designed a fitness function that rewarded speed and distance traveled, while heavily penalizing collisions. This creates an evolutionary pressure for the agent to drive fast but safely.

Technical Outcomes

23 Track Spline Points
Auto Self-Optimizing Path

Key Learning: This project taught me the basics of Machine Learning in Unreal engine. it was interesting to see how different inputs and outputs can affect the overall outcome.

Cooking With Words: Gen AI Integration

AI Game Jam - Collaborative Project

Role Co-Developer
Engine Unity (C#)
Tech LLM & Stable Diffusion
Type Game Jam

The Challenge

The challenge of the AI and Games Summer School jam was to integrate AI into a game in some way, myself and another developer chose to integreate GenAI into the game loop. Technically, the main hurdle was Latency. Generating an image with Stable Diffusion and text with an LLM takes seconds far too long to pause a game. We needed a way to process these heavy requests locally without freezing the main game thread.

The Solution

We designed an Asynchronous Gameplay Loop where the "Wait Time" became the "Game Time."

  • The Mechanic: The player acts as the "Prompt Engineer." By running between stations and typing ingredients, they are constructing the "Recipe" for the AI.
  • Pipeline: Once submitted, the data is sent to a locally running AI model. The game continues running while the LLM generates a description and Stable Diffusion generates the texture.
  • Runtime Assets: When the API returns, we convert the raw data into a Unity Texture2D and apply it to the UI cake pciture in real-time.

Technical Outcomes

Local Zero-Latency Cloud Dependency
Runtime Dynamic Texture Generation

Key Learning: This project taught me how to bridge C# (Unity) with Local API calls. It also highlighted the importance of error handling what happens if the AI generates something invalid?

Endless Runner: Mobile Optimization

Personal Mobile Project

Role Solo Developer
Engine Unreal Engine 5
Platform Android / Mobile
Focus Serialization & UI

The Challenge

The goal was to understand Unreal Engine better when first begining to use it, I used this opportunity create to an Endless runner designed to work on mobile devices. Creating an "Endless" game presents a unique memory challenge: spawning and destroying track segments infinitely will eventually cause garbage collection spikes (lag). I needed a way to maintain a smooth framerate while handling persistent data (High Scores) across sessions.

The Solution

I focused on efficient memory management and persistent systems:

  • Object Pooling Pattern: Instead of having pre constructed track pieces, I procedurally generate each track piece before the player reaches it using pooling so each track piece has a different obstacle configuration.
  • Serialization (Save System): Implemented a `USaveGame` subsystem to write binary data to the device, storing total Coin Count and High Scores between app launches.
  • Vector Math Mechanics: The "Magnet" upgrade uses vector interpolation (VInterp) to calculate the direction and velocity required to pull coin actors toward the player dynamically.

Technical Outcomes

60fps Target on Mobile Device
0 Runtime Allocations (Pooling)

Key Learning: This project was my first deep dive into Touch Input Interfaces. Mapping swipe gestures to lane changes required a different approach than standard keyboard input, using delta vectors to determine swipe direction.
